# OCI RAG Pipeline Resource Manager Stack

Terraform stack for OCI Resource Manager that provisions the foundational infrastructure for the `rag_pipeline` service:

- VCN, public and private subnets
- NSGs, internet/NAT/service gateways, route tables
- Public jump host VM for private administration
- Private Autonomous Database
- Private application VM
- Public API Gateway exposing the browser UI plus application APIs
- Vault secret for the generated application schema password

## What This Stack Produces

After a successful apply, the stack outputs:

- public API Gateway base URL
- public chat endpoint
- public health endpoint
- app VM OCID and private IP
- jump host public and private IPs
- Autonomous Database OCID, name, and private endpoint details
- generated application schema name
- generated application schema password secret name and OCID

## Scope

This workspace is intentionally separate from the runnable application source in:

- `../rag_pipeline`

The stack assumes the application code is delivered to the VM at bootstrap time from an external source:

- a Git repository, or
- a tarball/zip URL

## Current design choices

- API Gateway is public and does not enforce authentication.
- VM and Autonomous Database stay private.
- A small public jump host VM provides SSH access into the private VCN.
- The ADB admin password is expected to exist already in OCI Vault.
- The application schema password is generated by Terraform and stored in OCI Vault.
- The VM uses instance principals for OCI Generative AI calls and for reading the existing ADB admin password secret during bootstrap.
- Cloud-init bootstraps the VM and patches the downloaded app to use OCI instance principals when no OCI config file is present.
- The database must be provisioned on a vector-capable Autonomous AI Database version. This stack now pins `adb_db_version` explicitly instead of relying on the OCI default.

## Prerequisites

Before running this stack, the customer should already have:

1. A target compartment.
2. An OCI Vault and KMS key that can be used to store the generated app schema password secret.
3. A Vault secret containing the Autonomous Database admin password.
4. An Oracle Linux image OCID for the target region.
5. An application source URL:
   - public/private Git clone URL, or
   - downloadable tar.gz / zip archive URL

## Resource Manager Console Flow

1. Create a new stack in OCI Resource Manager.
2. Upload the sanitized stack zip from this folder.
3. Review the `schema.yaml` driven form.
4. Let Resource Manager auto-populate:
   - `tenancy_ocid`
   - `compartment_ocid`
   - `region`
5. Provide the remaining required inputs:
   - `home_region`
   - `vault_ocid`
   - `kms_key_ocid`
   - `adb_admin_password_secret_ocid`
   - `instance_image_ocid`
   - `ssh_public_key`
   - `app_source_type`
   - `app_source_url`
   - `app_source_ref`
6. Run `Plan`.
7. Review the planned infrastructure.
8. Run `Apply`.

## Notes

- This workspace remains separate from the runnable app repo so the current working application is not disturbed.
- The VM bootstrap creates or rotates the application schema, starts the service, and then delegates table creation to the application's `/api/bootstrap` endpoint.
- The stack outputs the generated schema secret name and OCID, not the raw password value.
- Tenancy IAM resources such as dynamic groups and policies are created through the tenancy home region provider.
- The RAG schema uses Oracle vector columns, so a 19c Autonomous Database is not sufficient for this application.
- The public API Gateway is configured to route `/`, `/governance`, `/ui/*`, `/assets/*`, `/api/*`, `/chat`, and `/health`.

## Packaging For Resource Manager

When creating the Resource Manager upload bundle:

- include the Terraform files, templates, lockfile, and example tfvars
- include `schema.yaml` so the OCI Console renders the guided form
- exclude `.terraform/`, `terraform.tfstate*`, and any environment-specific `*.auto.tfvars`
- provide customer-specific values through the Resource Manager stack variables UI instead of bundling them into the zip

## How To Build The Upload Zip

The upload zip for OCI Resource Manager should contain the contents of `rag_pipeline_stack/` at the root of the archive.

### From a local clone

1. Open the `rag_pipeline_stack/` folder.
2. Create a zip from that folder's contents only.
3. Verify the zip includes files such as:
   - `provider.tf`
   - `variables.tf`
   - `outputs.tf`
   - `schema.yaml`
   - `templates/bootstrap_app.sh.tftpl`
4. Verify the zip does not include:
   - `.terraform/`
   - `terraform.tfstate`
   - `terraform.tfstate.backup`
   - `customer.auto.tfvars`

### From GitHub

1. Download the repository source zip.
2. Extract it locally.
3. Open the extracted `rag_pipeline_stack/` folder.
4. Re-zip only the contents of `rag_pipeline_stack/`.
5. Upload that new zip to OCI Resource Manager.

## Post-Apply Validation

After apply completes:

1. Open the `api_gateway_base_url` output in a browser.
2. Open `${api_gateway_base_url}/governance`.
3. Upload a small `.txt` file.
4. Confirm `/health` returns `200`.
5. Ask a simple question from the main UI.
6. Confirm the schema password secret output exists in Vault.

## Pending IAM Step

If the deployment operator cannot create root-tenancy policies, a tenancy IAM admin must create this policy manually in the home region `eu-frankfurt-1`:

```text
Allow dynamic-group 'Default'/'<app_dynamic_group_name>' to use generative-ai-inference-family in compartment id <target_compartment_ocid>
Allow dynamic-group 'Default'/'<app_dynamic_group_name>' to read secret-bundles in compartment id <target_compartment_ocid>
```

After that policy exists, set `app_dynamic_group_enabled = false` for the final Terraform reconciliation if you do not want Terraform to manage the tenancy IAM objects.
